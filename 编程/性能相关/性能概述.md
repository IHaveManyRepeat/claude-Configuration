# 性能概述

性能优化是软件开发中的核心关注点，它直接影响用户体验、系统吞吐量和资源成本。性能优化不仅仅是让代码“更快”，而是需要在**正确性**、**可维护性**和**性能**之间找到最佳平衡点。

---

## 性能优化的核心目标

| 目标 | 描述 | 衡量指标 |
|------|------|----------|
| **响应时间** | 系统处理请求并返回结果的时间 | TTFB、首字节时间、页面加载时间 |
| **吞吐量** | 单位时间内系统能处理的请求数量 | QPS、TPS、并发数 |
| **资源利用率** | CPU、内存、磁盘、网络的使用效率 | CPU使用率、内存占用、IO等待 |
| **可扩展性** | 系统在高负载下的表现及扩展能力 | 垂直扩展比、水平扩展效率 |

---

## 提升代码性能的方法原则

### 1. 测量先行原则

**先测量，后优化。** 不以猜测为依据进行性能优化。

- 使用**性能分析工具**（Profiler）定位热点代码
- 建立**性能基准**（Baseline），确保优化有据可依
- 优化后与基准对比，验证优化效果
- 警惕**过早优化**，应优先保证代码正确性和可读性

> **反模式**：在未定位瓶颈的情况下“凭感觉”优化，认为“某种写法一定比另一种快”。

### 2. 二八法则

80%的性能问题通常由20%的代码造成。优化应聚焦于：

- **热点路径**：高频执行的代码路径
- **关键函数**：被大量调用的核心函数
- **数据密集**：处理大量数据的循环和算法

> **实践建议**：使用采样分析器（如 `perf`、`火焰图`）找出真实热点，而非依赖代码行数或主观判断。

### 3. 算法与数据结构优化

选择合适的算法和数据结构是性能优化的基础：

- **时间复杂度**：用更低复杂度的算法替代（如 O(n²) → O(n log n)）
- **空间换时间**：使用缓存、索引、预计算减少重复计算
- **数据局部性**：利用 CPU 缓存预取，提高数据访问效率
- **避免重复**：结果可复用时，使用记忆化（Memoization）或缓存

| 场景 | 低效做法 | 高效做法 |
|------|----------|----------|
| 查找 | 线性遍历 O(n) | 哈希表 O(1)、二分查找 O(log n) |
| 排序 | 冒泡排序 O(n²) | 快速排序、归并排序 O(n log n) |
| 字符串拼接 | 循环中 `+` 拼接 | StringBuilder、StringBuffer |
| 判重 | 每次遍历列表 | 哈希集合 O(1) |

### 4. 减少不必要的计算

- **惰性计算**：按需计算，避免预计算不常用的结果
- **剪枝**：在递归、搜索中跳过无效分支
- **短路求值**：利用 `&&`、`||` 的短路特性避免无效计算
- **避免重复计算**：提取循环不变式到循环外

### 5. 优化内存访问

内存访问模式对性能影响显著：

- **对象池**：复用对象，减少 GC 压力和分配开销
- **预分配**：对于已知大小的集合，预分配容量避免扩容
- **数据对齐**：利用 CPU 缓存行对齐提升访问效率
- **减少指针追踪**：连续内存访问优于分散的指针访问
- **及时释放**：及时释放大对象引用，协助垃圾回收

### 6. I/O 优化

I/O 操作通常是系统瓶颈的主要来源：

- **批量操作**：将多次 I/O 合并为一次（如批量写入、批量数据库操作）
- **异步 I/O**：利用异步非阻塞 I/O 提高并发能力
- **缓冲 I/O**：使用缓冲区减少系统调用次数
- **缓存**：利用内存缓存减少磁盘/网络 I/O
- **压缩**：对网络传输数据进行压缩，减少传输量
- **连接复用**：使用连接池复用数据库、网络连接

### 7. 并行与并发

充分利用多核 CPU 提升吞吐量：

- **并行计算**：将独立任务分配到多个核心（如 `parallel stream`、多线程）
- **异步编程**：利用事件循环处理 I/O 密集型任务
- **协程**：在语言支持的情况下使用轻量级协程
- **并行算法**：选择支持并行的算法和数据结构
- **避免竞争**：减少锁竞争，使用无锁数据结构

> **注意**：并行化会带来额外开销（线程创建、上下文切换、同步），只有在任务足够重时才有收益。

### 8. 缓存策略

合理使用缓存可以大幅提升性能：

- **多级缓存**：L1（CPU缓存）→ L2 → L3 → 内存 → 磁盘 → 网络
- **缓存失效策略**：LRU、TTL、Write-through、Write-back
- **缓存粒度**：按需选择缓存键的粗细程度
- **缓存击穿**：使用互斥锁或布隆过滤器防护
- **缓存雪崩**：使用随机过期时间、预热机制

### 9. 代码生成与预编译

- **预计算**：在编译期或启动时计算常量值
- **宏展开**：C/C++ 中使用宏减少运行时开销
- **JIT 编译**：利用即时编译优化热点代码
- **AOT 编译**： Ahead-of-Time 编译减少启动时间

### 10. 持续监控与回归防护

- **性能监控**：建立性能指标收集和告警机制
- **性能测试**：在 CI/CD 中集成性能测试，防止性能退化
- **预警阈值**：设置性能指标告警阈值，及早发现问题

## 性能优化的常见误区

| 误区 | 说明 |
|------|------|
| **过早优化** | 在未了解瓶颈的情况下优化，浪费精力且可能损害代码质量 |
| **微观优化忽视宏观** | 纠结于语法细节（如循环顺序）而忽视架构问题 |
| **迷信 benchmark** | 忽视真实场景，benchmark 结果不等于生产环境表现 |
| **优化他人代码** | 优先优化自己负责的模块，往往收益更大 |
| **一次性优化** | 性能优化是持续过程，不是一次性工作 |

## 性能优化检查清单

- [ ] 是否通过性能分析工具定位了真实瓶颈？
- [ ] 是否有性能基准作为对比依据？
- [ ] 选择的算法复杂度是否最优？
- [ ] 是否存在不必要的重复计算？
- [ ] 内存分配和访问模式是否高效？
- [ ] I/O 操作是否已批量化和异步化？
- [ ] 是否合理使用缓存？
- [ ] 是否充分利用多核并行能力？
- [ ] 优化后是否有性能测试验证？
- [ ] 代码可读性和可维护性是否保持？
