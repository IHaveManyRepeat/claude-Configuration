# 安全事件响应

> 本文档为安全事件响应提供系统化的指导框架，涵盖从事件发现到事后改进的完整生命周期。

## 1. 概述

### 1.1 定义

**安全事件响应**（Security Incident Response）是指组织针对安全事件采取的有计划、有组织的响应行动，旨在：
- 快速检测和确认安全事件
- 最小化事件影响和损失
- 恢复正常业务运营
- 从事件中学习并改进防御能力

### 1.2 重要性

| 重要性维度 | 说明 |
|-----------|------|
| 降低损失 | 快速响应可将数据泄露、系统破坏等损失降至最低 |
| 合规要求 | 满足 GDPR、PCI-DSS、网络安全法等法规的事件报告要求 |
| 品牌保护 | 及时、透明的事件处理维护客户信任 |
| 持续改进 | 通过事件分析发现安全弱点，完善防御体系 |

## 2. 安全事件分类

### 2.1 按事件类型分类

```
安全事件
├── 外部攻击类
│   ├── DDoS 攻击
│   ├── 恶意软件感染
│   ├── 钓鱼攻击
│   └── 零日漏洞利用
├── 内部威胁类
│   ├── 员工误操作
│   ├── 权限滥用
│   └── 恶意内部人员
├── 数据安全类
│   ├── 数据泄露
│   ├── 数据篡改
│   └── 数据丢失
└── 系统安全类
    ├── 未授权访问
    ├── 服务中断
    └── 配置错误
```

### 2.2 按影响级别分类

| 级别 | 描述 | 示例 | 响应时限 |
|------|------|------|----------|
| **P0 - 紧急** | 业务完全中断或大规模数据泄露 | 勒索软件锁定全部系统、核心数据库泄露 | 15 分钟 |
| **P1 - 高** | 核心业务受损或敏感数据受影响 | 部分系统被加密、客户信息泄露 | 1 小时 |
| **P2 - 中** | 非核心业务受影响 | 边缘系统被入侵、有限数据访问 | 4 小时 |
| **P3 - 低** | 潜在风险或轻微影响 | 可疑登录尝试、配置偏差告警 | 24 小时 |

## 3. 安全事件响应团队（CSIRT）

### 3.1 团队职责

| 角色 | 主要职责 |
|------|----------|
| **CSIRT 负责人** | 统筹协调、决策审批、对接管理层 |
| **安全分析师** | 事件检测、分析、分类和初步响应 |
| **应急响应工程师** | 技术层面的遏制、根除和恢复操作 |
| **法务顾问** | 法律合规、证据保全、监管报告 |
| **公关/沟通** | 内部沟通、对外声明、客户通知 |
| **IT 运维** | 系统恢复、基础设施支持 |

### 4.2 各阶段详细说明

#### 阶段 1：准备（Preparation）

**目标**：建立响应能力，确保事件发生时能快速响应

**关键活动**：

- [ ] 建立 CSIRT 团队，明确角色和职责
- [ ] 制定安全事件响应预案（IRP）
- [ ] 部署安全监控工具（SIEM、EDR、NDR）
- [ ] 建立事件分类和分级标准
- [ ] 准备应急响应工具包（取证工具、隔离脚本）
- [ ] 定期培训和演练

**交付物**：
- 安全事件响应预案
- 事件分类分级标准
- 应急联系清单
- 工具和资源清单

#### 阶段 2：检测与分析（Detection & Analysis）

**目标**：快速识别和确认安全事件，确定其范围和影响

**关键活动**：

1. **事件检测**
   - 分析安全告警（SIEM、IDS/IPS、EDR）
   - 审查日志异常
   - 接收用户报告
   - 威胁情报关联

2. **事件分析**
   - 确认事件真实性（排除误报）
   - 确定事件类型和攻击向量
   - 评估影响范围和严重程度
   - 收集关键证据（截图、日志、内存_dump）

3. **事件记录**
   - 记录事件时间线
   - 文档化 IOC（妥协指标）
   - 保存相关证据

**事件分析检查清单**：

```
□ 事件发生时间？
□ 受影响的系统/应用/数据？
□ 攻击者的入口点？
□ 攻击使用的技术手段？
□ 事件当前状态（活跃/已遏制）？
□ 影响的数据类型和数量？
□ 是否有数据外泄？
□ 需要通知哪些相关方？
```

**沟通升级标准**：

| 条件 | 升级对象 |
|------|----------|
| 确认数据泄露 | 法务、公关、高管 |
| 影响关键业务 | 业务部门负责人 |
| 涉及执法调查 | 法务、外部律师 |
| 媒体关注风险 | 公关、PR |

#### 阶段 3：遏制（Containment）

**目标**：阻止事件进一步扩散，控制损害范围

**短期遏制措施（立即执行）**：

- [ ] 隔离受感染系统（网络隔离/断网）
- [ ] 禁用被利用的账户
- [ ] 阻断恶意 IP/域名
- [ ] 暂停受影响的服务
- [ ] 阻断攻击路径

**长期遏制措施（短期遏制后）**：

- [ ] 账户密码重置
- [ ] 系统安全加固
- [ ] 增强监控规则
- [ ] 部署额外安全控制

**遏制原则**：

| 原则 | 说明 |
|------|------|
| 平衡业务 | 遏制措施不应造成比事件本身更大的业务影响 |
| 保留证据 | 在采取行动前尽可能保存证据 |
| 最小权限 | 仅禁用必要的系统和账户 |

#### 阶段 4：根除（Eradication）

**目标**：彻底消除系统中存在的威胁，防止复发

**关键活动**：

- [ ] 识别并消除根本原因
- [ ] 删除恶意软件/后门
- [ ] 修复漏洞
- [ ] 重置受影响的账户密码
- [ ] 重装受感染系统（如必要）
- [ ] 确保无残留威胁

**验证标准**：
- 系统已更新到安全版本
- 无恶意进程/文件
- 无后门或持久化机制
- 监控未发现异常

#### 阶段 5：恢复（Recovery）

**目标**：安全地将系统恢复到正常运营状态

**关键活动**：

1. **系统恢复**
   - 从干净备份恢复数据
   - 验证系统完整性
   - 逐步恢复服务（从低风险到高风险）

2. **验证测试**
   - 功能测试
   - 安全测试
   - 渗透测试（如需要）

3. **持续监控**
   - 加强日志监控
   - 设置更严格的告警
   - 观察是否有复发迹象

**恢复后检查清单**：

```
□ 系统功能正常运行？
□ 数据完整性验证通过？
□ 无异常网络连接？
□ 无异常进程？
□ 监控告警已更新？
□ 用户可正常访问？
```

#### 阶段 6：事后分析（Post-Incident Analysis）

**目标**：从事件中学习，改进防御能力

**关键活动**：

1. **事件复盘**
   - 回顾事件时间线
   - 分析根本原因
   - 评估响应过程
   - 识别改进点

2. **文档更新**
   - 更新事件响应预案
   - 完善检测规则
   - 补充知识库

3. **修复措施**
   - 实施安全改进
   - 加强员工培训
   - 优化流程

**事后分析报告模板**：

```

## 事件概述
```

## 5. 应急响应预案模板

### 5.1 勒索软件响应预案

```
# 勒索软件事件响应预案

## 触发条件
系统文件被加密，出现勒索信息，或检测到勒索软件特征。

## 响应步骤

### 第 1 步：立即隔离（15 分钟内）
- [ ] 断开受感染系统的网络连接
- [ ] 禁用网络共享
- [ ] 隔离可能受感染的其他系统
- [ ] 通知 CSIRT 团队

### 第 2 步：评估范围（30 分钟内）
- [ ] 确定受影响的系统数量
- [ ] 识别加密文件的类型（业务数据、个人信息）
- [ ] 检查是否有数据备份

### 第 3 步：遏制
- [ ] 关闭不必要的服务
- [ ] 阻断勒索软件可能的传播途径
- [ ] 收集恶意软件样本

### 第 4 步：恢复
- [ ] 评估是否支付赎金（不推荐）
- [ ] 从备份恢复（确认备份未被感染）
- [ ] 逐步恢复系统运行

### 第 5 步：通知
- [ ] 内部通知相关方
- [ ] 评估是否需要向监管部门报告
- [ ] 如涉及个人信息，通知受影响用户

## 决策点
- 是否需要执法机构介入？
- 是否需要外部专家支持？
- 是否需要通知保险？
```

### 5.2 数据泄露响应预案

```
# 数据泄露事件响应预案

## 触发条件
确认或高度怀疑敏感数据（个人信息、商业机密）被未授权访问或外泄。

## 响应步骤

### 第 1 步：确认泄露（1 小时内）
- [ ] 确认数据泄露的真实性
- [ ] 确定泄露数据的类型和范围
- [ ] 评估受影响的数据主体数量

### 第 2 步：遏制泄露
- [ ] 关闭数据泄露的入口点
- [ ] 禁用被滥用的账户
- [ ] 阻断数据外传通道

### 第 3 步：评估影响
- [ ] 数据敏感性分析
- [ ] 受影响用户风险评估
- [ ] 法律和合规影响评估

### 第 4 步：通知
- [ ] 内部通知（管理层、法务、公关）
- [ ] 监管机构通知（根据法规要求）
- [ ] 受影响用户通知（GDPR 72 小时內）

### 第 5 步：补救
- [ ] 提供信用监控服务（如涉及个人信息）
- [ ] 账户安全建议
- [ ] 修复漏洞，防止再次泄露
```

## 6. 沟通与报告机制

### 6.1 内部沟通

| 沟通对象 | 沟通内容 | 时限 | 沟通方式 |
|----------|----------|------|----------|
| CSIRT 团队 | 事件详情、响应行动 | 实时 | 应急群/电话 |
| 管理层 | 影响评估、所需资源 | 30 分钟内 | 电话/会议 |
| 业务部门 | 影响范围、配合事项 | 1 小时内 | 邮件/会议 |
| 全员（如必要） | 事件说明、行动指引 | 视情况 | 邮件/公告 |

### 6.2 外部沟通

| 沟通对象 | 沟通内容 | 时限 | 审批人 |
|----------|----------|------|--------|
| 监管部门 | 事件报告（如需） | 按法规 | 法务/高管 |
| 客户 | 事件说明、补救措施 | 视影响 | 公关/法务 |
| 媒体（如需） | 官方声明 | 视情况 | 公关/高管 |
| 执法机构 | 证据提供、配合调查 | 视情况 | 法务 |

### 6.3 沟通原则

1. **及时性**：快速响应，但确保信息准确
2. **准确性**：只提供经过核实的信息
3. **一致性**：对外口径统一，由专人统一回应
4. **保密性**：敏感信息只在需要知道的人员中共享

## 9. 合规要求

### 9.1 主要法规要求

| 法规 | 事件报告要求 |
|------|--------------|
| 《网络安全法》 | 72 小时内向监管部门报告 |
| GDPR | 72 小时内通知监管机构 |
| PCI-DSS | 立即通知卡组织 |
| 《个人信息保护法》 | 及时通知个人信息主体 |

### 9.2 证据保全

- 保留所有相关日志（至少 6 个月）
- 记录完整的取证 chain of custody
- 如涉及执法调查，协调取证过程
- 咨询法务确保证据合规性

## 10. 持续改进

### 10.1 改进机制

1. **每次事件后**：进行复盘，记录改进点
2. **每季度**：评估响应指标，修订预案
3. **每年**：全面审查，更新流程和工具
4. **持续**：关注行业最佳实践，更新知识库

### 10.2 关键指标

| 指标 | 目标 |
|------|------|
| 平均检测时间（MTTD） | < 24 小时 |
| 平均响应时间（MTTR） | < 4 小时 |
| 事件复发率 | < 5% |
| 预案覆盖率 | 100% 常见事件类型 |
