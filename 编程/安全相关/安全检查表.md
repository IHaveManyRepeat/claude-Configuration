# 安全检查表

> 本检查表用于代码开发过程中的安全验证，确保系统具备高安全性。
> 每次代码提交前应逐项核对，重要功能上线前必须完成全面检查。

---

## 一、输入验证与过滤

### 1.1 通用输入验证
- [ ] 所有用户输入必须经过验证，不信任任何外部数据
- [ ] 使用白名单验证方式，优于黑名单验证
- [ ] 验证数据类型、长度、格式、范围
- [ ] 验证必填字段是否提供
- [ ] 验证文件上传的类型、大小、内容

### 1.2 注入防护
- [ ] **SQL 注入**：所有数据库查询使用参数化查询或 ORM
- [ ] **NoSQL 注入**：使用安全查询构建器，避免字符串拼接
- [ ] **命令注入**：避免使用用户输入执行系统命令，必要时使用白名单
- [ ] **XSS（跨站脚本）**：所有用户输出进行适当转义或使用安全渲染框架
- [ ] **LDAP 注入**：使用参数化查询或输入转义

---

## 二、身份认证与授权

### 2.1 认证安全
- [ ] 密码使用强哈希算法存储（bcrypt、argon2、PBKDF2）
- [ ] 密码复杂度要求：至少 8 位，包含大小写字母、数字、特殊字符
- [ ] 登录失败实施账户锁定策略（5 次失败后锁定 15-30 分钟）
- [ ] 实现多因素认证（MFA）支持
- [ ] 会话令牌使用安全随机数生成，长度 ≥ 128 位
- [ ] 敏感操作需要重新认证
- [ ] OAuth 2.0 / OIDC 使用安全配置，验证 state 参数防止 CSRF

### 2.2 授权控制
- [ ] 实现基于角色的访问控制（RBAC）或基于属性的访问控制（ABAC）
- [ ] 遵循最小权限原则
- [ ] 验证用户是否有权访问请求的资源
- [ ] 防止垂直越权（普通用户无法访问管理员功能）
- [ ] 防止水平越权（用户无法访问其他用户的数据）
- [ ] API 端点实施速率限制

---

## 三、数据保护

### 3.1 敏感数据处理
- [ ] 敏感数据（密码、密钥、身份证号等）加密存储
- [ ] 敏感数据在日志中脱敏，不记录明文密码、完整信用卡号等
- [ ] 敏感数据在内存中处理后及时清除
- [ ] 数据传输使用 TLS 1.2+ 加密
- [ ] 密钥管理使用专用密钥管理服务（KMS），不硬编码密钥

### 3.2 隐私保护
- [ ] 遵守 GDPR、CCPA 等隐私法规
- [ ] 提供数据删除功能（被遗忘权）
- [ ] 用户数据收集最小化原则
- [ ] 隐私政策清晰说明数据用途

---

## 四、API 安全

### 4.1 请求安全
- [ ] 所有 API 使用 HTTPS
- [ ] 实施 API 密钥或 JWT 认证
- [ ] 验证请求头的 Content-Type
- [ ] 限制请求体大小，防止 DoS 攻击
- [ ] CORS 配置使用白名单域名
- [ ] CSRF 令牌保护状态修改操作

### 4.2 响应安全
- [ ] 响应不泄露服务器版本信息
- [ ] 响应不泄露敏感配置信息
- [ ] 错误信息通用化，不暴露内部实现细节
- [ ] 安全头正确设置（X-Content-Type-Options、X-Frame-Options 等）

---

## 五、认证与会话管理

### 5.1 会话安全
- [ ] 会话 ID 在登录后重新生成
- [ ] 会话超时机制合理配置（ inactivity 15-30 分钟）
- [ ] 安全的会话 cookie 属性：HttpOnly、Secure、SameSite
- [ ] 防止会话固定攻击
- [ ] 并发登录控制（可选）

### 5.2 Token 安全
- [ ] JWT 使用非对称签名（RS256）
- [ ] JWT 设置合理的过期时间（access token ≤ 1 小时）
- [ ] JWT 存储在安全位置，避免 XSS 窃取
- [ ] 刷新令牌安全存储，定期轮换

---

## 六、加密与密钥管理

### 6.1 加密算法
- [ ] 对称加密使用 AES-256-GCM 或 ChaCha20-Poly1305
- [ ] 非对称加密使用 RSA-2048+ 或 ECC（ Curve25519）
- [ ] 哈希算法使用 bcrypt、argon2 或 PBKDF2
- [ ] 不使用已知不安全的算法（MD5、SHA1 用于安全目的、DES、3DES）
- [ ] 随机数生成使用安全随机数生成器

### 6.2 密钥管理
- [ ] 密钥不硬编码在代码中
- [ ] 密钥存储在环境变量或专用密钥管理服务
- [ ] 密钥轮换机制（定期更换）
- [ ] 密钥分离：不同环境使用不同密钥

---

## 七、错误处理与日志

### 7.1 错误处理
- [ ] 错误信息不泄露敏感数据（堆栈信息、文件路径、配置）
- [ ] 自定义错误页面，避免暴露框架信息
- [ ] 全局异常处理器捕获未预期异常
- [ ] 失败时安全降级，不暴露系统细节

### 7.2 安全日志
- [ ] 记录登录失败事件
- [ ] 记录权限拒绝事件
- [ ] 记录敏感操作（如数据删除、配置修改）
- [ ] 记录异常和错误
- [ ] 日志中不记录敏感信息（密码、密钥、完整信用卡号）
- [ ] 日志保留足够长时间（至少 6 个月）
- [ ] 日志完整性保护（防篡改）

---

## 八、依赖与组件安全

### 8.1 依赖管理
- [ ] 定期更新依赖库到安全版本
- [ ] 使用工具扫描已知漏洞（OWASP Dependency-Check、Snyk）
- [ ] 移除未使用的依赖
- [ ] 验证第三方包的完整性和来源

### 8.2 安全配置
- [ ] 生产环境禁用调试模式
- [ ] 生产环境禁用详细错误页面
- [ ] 删除或保护调试端点
- [ ] 安全配置检查清单

---

## 九、网络与基础设施安全

### 9.1 传输安全
- [ ] 全站使用 HTTPS
- [ ] 使用 HSTS 头
- [ ] 证书有效且使用安全配置
- [ ] 不使用自签名证书（生产环境）

### 9.2 防御措施
- [ ] 实施 Web 应用防火墙（WAF）
- [ ] DDoS 防护措施
- [ ] IP 白名单/黑名单控制
- [ ] 安全的文件上传目录无执行权限

## 十、常见漏洞检查

### 10.1 OWASP Top 10 检查
- [ ] **A01 访问控制失效**：权限验证完整
- [ ] **A02 加密失败**：敏感数据加密
- [ ] **A03 注入**：无 SQL/NoSQL/命令注入
- [ ] **A04 不安全设计**：安全架构设计
- [ ] **A05 安全配置错误**：安全配置检查
- [ ] **A06 易受攻击组件**：依赖更新
- [ ] **A07 识别和身份验证失败**：强认证
- [ ] **A08 软件和数据完整性失败**：完整性验证
- [ ] **A09 日志记录和监控不足**：安全日志
- [ ] **A10 服务器端请求伪造（SSRF）**：验证 URL 来源

### 10.2 其他常见漏洞
- [ ] **目录遍历**：文件路径验证
- [ ] **XML 外部实体（XXE）**：禁用或安全配置 XML 解析器
- [ ] **反序列化漏洞**：安全的反序列化实践
- [ ] **竞赛条件**：并发安全

### 11.2 配置验证

- [ ] 密钥已更新为生产密钥
- [ ] 安全头已正确配置
- [ ] 日志级别已调整为生产级别
- [ ] 监控告警已配置

## 使用说明

1. **开发阶段**：每完成一个功能模块，对照检查
2. **Code Review**：提交代码前由另一位开发者复查
3. **上线前**：安全团队进行全面检查
4. **定期审计**：每季度进行安全审计
