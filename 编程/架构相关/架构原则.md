# 架构原则

> 设计高可维护、可扩展系统的核心原则

---

## 1. 依赖倒置原则 (DIP)

**高层模块不应该依赖低层模块，两者都应该依赖抽象**。

```go
// 坏：高层依赖低层
type UserService struct {
    db *mysql.DB
}

// 好：依赖抽象
type UserRepository interface {
    FindByID(id string) (*User, error)
    Save(user *User) error
}

type UserService struct {
    repo UserRepository  // 依赖接口，不依赖具体实现
}
```

---

## 2. 开放封闭原则 (OCP)

**对扩展开放，对修改封闭**。新增功能通过扩展而非修改现有代码实现。

```go
// 坏：每次新增支付方式都要修改这个函数
func Pay(amount float64, method string) {
    if method == "alipay" { /* ... */ }
    if method == "wechat" { /* ... */ }
    // 新增一个就要改这里
}

// 好：新增支付方式只需添加新实现
type Payment interface { Pay(amount float64) error }
type Alipay struct{}
type WechatPay struct{}
// 新增 ApplePay 不用改现有代码
```

---

## 3. 里氏替换原则 (LSP)

**子类型必须能够替换其基类型**。父类出现的地方，子类也能出现。

```go
// 坏：子类行为与父类不一致
type Reader interface { Read() string }
type FileReader struct{}
type NetworkReader struct{}

// NetworkReader.Read() 在超时时返回空字符串，导致调用方崩溃
```

---

## 4. 接口隔离原则 (ISP)

**客户端不应该依赖它不需要的接口**。接口应该小而专。

```go
// 坏：胖接口，什么都要
type Machine interface {
    Print()
    Scan()
    Fax()
}

// 好：拆成小接口
type Printer interface { Print() }
type Scanner interface { Scan() }
// 按需依赖
```

---

## 5. 单一职责原则 (架构层面)

**每个模块、每个服务、每个类只有一个变化原因**。

```
// 好的服务划分
├── user-service      // 只管用户
├── order-service     // 只管订单
├── notification-service  // 只管通知
```

---

## 6. 关注点分离 (Separation of Concerns)

**把不同层面的关注点分开**。

```
// 典型分层架构
├── controller  // 处理 HTTP 请求
├── service     // 处理业务逻辑
├── repository  // 处理数据访问
└── model       // 数据模型
```

---

## 7. 领域驱动设计 (DDD)

**用业务领域语言建模，边界清晰的领域模型**。

```
// 订单上下文
Order aggregate
├── Order (根实体)
│   ├── OrderItems (值对象集合)
│   └── ShippingAddress (值对象)
├── OrderRepository (接口)
└── OrderService (领域服务)
```

---

## 8. KISS 原则 (架构)

**架构也要简单**。不要为了"未来扩展"而过度设计。

- 优先选择最简单的方案
- 不要为了扩展性而增加不必要的抽象层
- YAGNI：现在不需要的架构，就不要现在做

---

## 9. YAGNI 原则 (架构)

**只实现当前明确所需的功能**。

```go
// 坏：预留给"将来可能的需求"
type UserService struct {
    currentPlan string  // 暂时不用，先预留
    features    map[string]bool  // 将来可能加功能
}

// 好：只做现在需要的
type UserService struct {
    repo UserRepository
}
```

---

## 10. 事件驱动架构 (EDA)

**通过事件解耦系统**。

```go
// 发布事件
orderService.PlaceOrder(ctx, order) {
    repo.Save(order)
    eventBus.Publish(OrderPlacedEvent{OrderID: order.ID})
}

// 订阅处理
eventBus.Subscribe("order.placed", func(e Event) {
    notificationService.SendEmail(e.OrderID)
})
```

---

## 11. 微服务边界

**按业务边界划分微服务**。

- 每个服务独立部署、独立扩展
- 服务间通过 API 通信，不共享数据库
- 避免distributed monolith（分布式单体）

---

## 12. CQRS 模式

**命令查询职责分离**。

- Command (写)：修改状态
- Query (读)：不修改状态，返回数据

```
写入模型 → 同步/异步 → 读取模型
    (优化写入)          (优化读取)
```

---

## 13. 六边形架构 (Ports & Adapters)

**核心业务逻辑与外部依赖完全解耦**。

```
        ┌─────────────┐
        │   Adapter   │  ← HTTP, CLI, Message Queue
        └──────┬──────┘
               ↓
        ┌─────────────┐
        │    Port     │  ← 接口定义
        └──────┬──────┘
               ↓
        ┌─────────────┐
        │   Domain    │  ← 核心业务逻辑
        │   (纯逻辑)   │
        └──────┬──────┘
               ↓
        ┌─────────────┐
        │    Port     │
        └──────┬──────┘
               ↓
        ┌─────────────┐
        │   Adapter   │  ← MySQL, Redis, External API
        └─────────────┘
```

## 14. 熔断与降级

**防止级联故障**。

```go
// 熔断器示例
type CircuitBreaker struct {
    failures    int
    threshold   int
    state       string  // closed, open, half-open
}

func (cb *CircuitBreaker) Execute(fn func()) error {
    if cb.state == "open" {
        return ErrCircuitOpen  // 快速失败
    }
    err := fn()
    if err != nil {
        cb.failures++
        if cb.failures > cb.threshold {
            cb.state = "open"
        }
    }
    return err
}
```

## 15. 缓存策略

**正确使用缓存**。

- Cache-Aside：应用自行管理缓存
- Write-Through：同步写缓存和数据库
- Write-Behind：异步写数据库
- TTL：设置过期时间
- 一致性：缓存与数据库的一致性策略
