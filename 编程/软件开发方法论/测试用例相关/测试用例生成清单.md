# 测试用例生成清单

> **定位**：测试用例设计时的实操检查清单。逐项确认覆盖完整性，避免遗漏。配合 [[测试用例生成方法]] 使用。

#软件开发方法论/TDD #测试用例设计 #检查清单

---

## 一、需求分析阶段

### 1.1 业务规则确认

- [ ] 所有业务规则已列出（正常规则 + 异常规则）
- [ ] 规则的前置条件已明确
- [ ] 规则的触发条件已明确
- [ ] 规则的优先级/冲突处理已明确
- [ ] 隐式需求已挖掘（性能、安全、兼容性）

### 1.2 输入输出分析

- [ ] 所有输入参数已识别
- [ ] 每个参数的数据类型已确认
- [ ] 每个参数的取值范围已确认
- [ ] 参数间的依赖关系已确认
- [ ] 输出结果的所有可能值已列举

---

## 二、用例设计阶段

### 2.1 等价类划分

- [ ] 每个输入的有效等价类已划分
- [ ] 每个输入的无效等价类已划分
- [ ] 有效等价类已组合覆盖（一个用例覆盖多个有效类）
- [ ] 无效等价类单独覆盖（一个用例只含一个无效条件）

### 2.2 边界值分析

- [ ] 每个等价类的边界已识别
- [ ] 边界值（正好等于边界）已测试
- [ ] 边界-1（刚低于边界）已测试
- [ ] 边界+1（刚高于边界）已测试
- [ ] 特殊边界已覆盖：0、空字符串、空数组、null/undefined
- [ ] 数值上下限已覆盖：最大值、最小值、溢出值

### 2.3 组合覆盖

- [ ] 参数组合场景已分析
- [ ] 全量组合数量已评估（是否可接受）
- [ ] 组合数过大时已使用正交表/成对测试缩减
- [ ] 关键组合（高风险）已额外保留（即使正交表未覆盖）
- [ ] 互斥/依赖条件已正确处理

### 2.4 场景覆盖

- [ ] 主流程（Happy Path）已覆盖
- [ ] 备选流程（Alternative Flow）已覆盖
- [ ] 异常流程（Exception Flow）已覆盖
- [ ] 并发/竞态场景已考虑
- [ ] 回退/撤销场景已覆盖

---

## 三、特殊场景检查

### 3.1 异常与错误处理

- [ ] 网络超时 / 连接失败
- [ ] 服务不可用 / 熔断降级
- [ ] 数据库连接失败 / 慢查询
- [ ] 外部依赖（API/第三方服务）异常
- [ ] 输入数据格式错误 / 损坏数据
- [ ] 权限不足 / 未认证 / Token 过期
- [ ] 资源耗尽（内存、磁盘、连接池）

### 3.2 外部依赖故障场景

> 每个外部依赖都必须回答：**挂了怎么办？慢了怎么办？拒绝了怎么办？恢复后能自动恢复吗？**

- [ ] **数据库**：宕机时有降级方案吗？连接池耗尽时能快速失败吗？慢查询会拖垮服务吗？恢复后数据一致吗？
- [ ] **Redis / 缓存**：宕机时能降级直连数据库吗？有缓存穿透保护（布隆过滤器）吗？热点 Key 过期会雪崩吗？有互斥锁重建机制吗？
- [ ] **模型 API**：配额用完（429）时有限流排队吗？能降级到轻量模型吗？响应超时有重试+熔断吗？返回异常内容有校验拦截吗？
- [ ] **第三方 API**：服务不可用（503）时有兜底数据吗？认证 Token 过期能自动刷新吗？API 变更有版本兼容策略吗？
- [ ] **消息队列**：队列堆积有背压控制吗？消费者宕机消息会丢吗？重复消费是幂等的吗？
- [ ] **对象存储 / 文件服务**：上传超时有断点续传吗？文件不存在有优雅 404 吗？存储满时有降级策略吗？
- [ ] **网络**：DNS 解析失败有本地缓存吗？完全断连有离线模式吗？高延迟/丢包有重试+幂等保证吗？
- [ ] **重试安全性**：所有重试操作是幂等的吗？重试不会导致重复扣款/重复发送吗？

### 3.3 安全测试

- [ ] SQL 注入（参数化查询验证）
- [ ] XSS（HTML 转义验证）
- [ ] CSRF（Token 验证）
- [ ] 越权访问（水平 / 垂直越权）
- [ ] 敏感数据泄露（日志、响应、URL 参数）
- [ ] 文件上传漏洞（类型/大小/路径）
- [ ] 速率限制（Rate Limiting）

### 3.4 数据与状态

- [ ] 空数据（空表、空列表、无记录）
- [ ] 大数据量（分页边界、超长字符串、大文件）
- [ ] 并发写入（重复提交、幂等性）
- [ ] 数据一致性（事务回滚、分布式一致性）
- [ ] 时区与日期（跨时区、闰年、月末、午夜）
- [ ] 编码与字符（Unicode、Emoji、特殊字符 `'"<>&\`）

### 3.5 兼容性与环境

- [ ] 不同浏览器 / 客户端版本
- [ ] 不同操作系统
- [ ] 不同屏幕尺寸 / 分辨率
- [ ] 不同 API 版本（向后兼容）
- [ ] 国际化 / 多语言

### 3.6 对抗性测试

- [ ] 模糊测试：随机/半随机输入是否导致崩溃
- [ ] 超长字符串 / 超大数值 / 嵌套极深 JSON 是否被正确处理
- [ ] 控制字符（`\x00`-`\x1F`）和不可见字符是否被过滤
- [ ] 注入类攻击（SQL / XSS / Command / LDAP / Template）是否被防御
- [ ] 路径穿越（`../`、`..\\`）是否被拦截
- [ ] 序列化攻击（反序列化炸弹、原型污染）是否被防御
- [ ] 类型混淆（字符串传给数字参数、数组传给对象参数）是否被校验

### 3.7 AI 系统对抗测试

- [ ] Prompt 注入：系统提示是否可被用户输入覆盖
- [ ] 角色劫持：模型角色设定是否可被重新定义
- [ ] 越狱测试：恶意请求是否被安全策略拦截
- [ ] 编码绕过：Base64/ROT13 混淆的恶意指令是否被检测
- [ ] 多轮对话：安全策略是否在长对话中持续生效
- [ ] 幻觉检测：模型对未知事实是否承认不知道
- [ ] 输出过滤：响应是否泄露系统内部信息
- [ ] 格式遵循：严格格式要求是否被 100% 遵守

## 四、用例质量检查

### 4.1 TDD 实践检查（必要项）

- [ ] **测试先行**：先写失败测试，再写实现代码（Red → Green → Refactor）
- [ ] **AAA 结构**：每个测试严格遵循 Arrange-Act-Assert 三段式
- [ ] **命名规范**：测试名描述行为（`should X when Y`），不描述实现细节
- [ ] **单一行为**：一个测试只验证一个行为/一个场景
- [ ] **独立性**：每个测试可独立运行，无执行顺序依赖
- [ ] **无共享可变状态**：测试间不共享可修改的数据
- [ ] **Mock 最小化**：只 Mock 外部依赖，不 Mock 被测系统本身
- [ ] **精确断言**：断言期望的具体值，避免 `expect.any()` / `expect.anything()`
- [ ] **路径覆盖 100%**：所有执行路径（正常路径 + 异常路径）必须被测试覆盖

### 4.2 TDD 实践检查（推荐项）

- [ ] **Happy Path + Sad Path**：正常路径和错误路径都有覆盖
- [ ] **边界值覆盖**：每个输入参数的边界值至少一个测试
- [ ] **空值/异常输入**：null、undefined、空字符串、空数组、零值
- [ ] **错误信息可读**：断言失败时的 message 能清晰表达期望
- [ ] **无 `skip` 遗留**：被 `skip` / `todo` 的测试不超过 5%，且附带原因注释
- [ ] **测试运行速度**：Unit < 10ms，Integration < 500ms，整体套件 < 2min

### 4.3 TDD 实践检查（加分项）

- [ ] **测试代码可读性**：测试代码和生产代码同等质量，无魔法数字
- [ ] **参数化测试**：相似逻辑的多个输入用参数化（`each` / `@pytest.mark.parametrize`）
- [ ] **工厂函数**：用工厂函数创建测试数据，避免重复 setup
- [ ] **快照测试适度**：仅在 UI 组件 / 序列化格式场景使用，不滥用
- [ ] **属性测试**：核心算法补充 property-based test

### 4.4 单条用例质量

- [ ] 用例标题描述**行为**而非实现
- [ ] 前置条件（Arrange）清晰明确
- [ ] 操作步骤（Act）只有一步核心操作
- [ ] 预期结果（Assert）精确、可验证
- [ ] 测试数据有明确的构造方式
- [ ] 不依赖其他用例的执行结果

### 4.5 用例集质量

- [ ] 无冗余用例（相同场景不重复）
- [ ] 无矛盾用例（期望结果一致）
- [ ] 用例优先级已标注（P0/P1/P2/P3）
- [ ] 自动化可行性已评估
- [ ] 用例总数在可控范围内（参考正交表压缩）

## 五、基于风险的覆盖确认

### 5.1 模块风险评估

| 模块 | 发生概率 | 影响程度 | 风险等级 | 覆盖策略 |
|------|----------|----------|----------|----------|
| （填写） | 高/中/低 | 高/中/低 | 关键/高/中/低 | 路径覆盖 100% |

### 5.2 覆盖度目标确认

- [ ] **所有模块统一目标：路径覆盖 100%**（正常路径 + 异常路径 + 边界路径必须全覆盖）
- [ ] 高风险模块额外要求：关键业务路径（支付、权限、账户安全等）必须有专项场景测试
- [ ] 整体用例预算分配合理（高 60% / 中 30% / 低 10%）

## 六、评审与复盘检查

### 6.1 用例评审

- [ ] 产品已确认业务场景完整性
- [ ] 开发已确认技术边界和异常分支
- [ ] 测试已确认覆盖度和可测试性
- [ ] 评审发现的遗漏已补充用例
- [ ] 评审争议点已记录并跟进

### 6.2 故障复盘（Bug → 用例）

- [ ] 线上 Bug 根因已分析
- [ ] 已编写对应的回归测试用例
- [ ] 用例已加入回归用例库
- [ ] 同类模块已排查是否存在类似风险
- [ ] 预防措施已落实到开发/测试流程
