# 重构原则精简版

> 来源：《重构：改善既有代码的设计》（Martin Fowler）
> 用途：Claude Code 编码时遵循的快速参考

---

## 核心理念（3 条）

1. **行为不变原则**：重构只改结构不改行为，每次小步修改后必须保证代码可工作
2. **两顶帽子原则**：添加功能时不重构，重构时不添加功能，二者不可混做
3. **三次法则**：第一次直接做，第二次勉强做，第三次必须重构

## 何时重构（4 条）

- **预备性重构**：添加新功能前，先整理代码使其易于扩展
- **理解性重构**：阅读代码时，边读边整理，让代码表达自己的意图
- **捡垃圾式重构**：顺手改进路过的坏代码，让营地比来时更干净
- **复审性重构**：代码审查时，边审边重构

## 识别坏味道 → 行动（24 条一句话速查）

### 命名/结构
| 坏味道 | 一句话判断标准 | 首选手法 |
|--------|---------------|---------|
| 神秘命名 | 名字不能立刻表达用途 → 改名 |
| 重复代码 | 同一段逻辑出现两次以上 → 提炼函数 |
| 过长函数 | 需要滚动才能看完 → 提炼函数、拆分阶段 |
| 过长参数列表 | 超过 3 个参数 → 引入参数对象 |

### 数据
| 坏味道 | 一句话判断标准 | 首选手法 |
|--------|---------------|---------|
| 全局数据 | 任何地方都能修改 → 封装变量 |
| 可变数据 | 数据被多处修改 → 封装变量、拆分变量 |
| 数据泥团 | 几项数据总是一起出现 → 提炼类 |
| 基本类型偏执 | 用原始类型代替有行为的对象 → 以对象取代基本类型 |

### 设计
| 坏味道 | 一句话判断标准 | 首选手法 |
|--------|---------------|---------|
| 发散式变化 | 一个类因不同原因在不同方向变化 → 拆分阶段、提炼类 |
| 霰弹式修改 | 一个变化要改多处 → 搬移函数、搬移字段 |
| 依恋情结 | 函数跟另一个类交流比跟自己的类还多 → 搬移函数 |
| 过大的类 | 一个类做了太多事 → 提炼类 |
| 内幕交易 | 模块间交换太多内部数据 → 隐藏委托关系 |
| 临时字段 | 字段只在某些时候有值 → 提炼类、引入特例 |

### 条件/继承
| 坏味道 | 一句话判断标准 | 首选手法 |
|--------|---------------|---------|
| 重复的 switch | 相同条件分支出现在多处 → 以多态取代条件表达式 |
| 循环语句 | 循环中混入复杂逻辑 → 以管道取代循环 |
| 被拒绝的遗承 | 子类只用了父类部分功能 → 以委托取代子类 |
| 夸夸其谈通用性 | 为"将来可能"而写的代码 → 删除它 |

### 其他
| 坏味道 | 一句话判断标准 | 首选手法 |
|--------|---------------|---------|
| 注释 | 需要注释来解释代码 → 先重构让代码自解释 |
| 冗赘的元素 | 函数/类没有增加信息量 → 内联它 |
| 纯数据类 | 只有字段没有行为 → 搬移行为进去 |
| 过长的消息链 | a.b().c().d() → 隐藏委托关系 |
| 中间人 | 类只做转发 → 移除中间人 |

## 编码时的重构心法

1. **小步前进**：每次只做一个小修改，改完跑测试，再改下一个
2. **测试先行**：没有自动化测试就不重构；重构前先确保测试通过
3. **消除重复**：重复是万恶之源，看到重复就提炼
4. **函数短小**：每个函数只做一件事，名字要说清楚这件事
5. **封装数据**：数据不可随意修改，通过函数控制访问
6. **用多态消灭 switch**：相同条件的 switch 出现两次以上，就用多态替换
7. **代码自解释**：好的代码不需要注释来解释"做了什么"，注释只说明"为什么"
8. **不为未来设计**：YAGNI——不为当前不需要的需求添加复杂性
